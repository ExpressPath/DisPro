# Dispro 実装フェーズ

## 全体方針

Dispro は、ユーザーから見ると「送るだけで処理される」サービスですが、内部では注文、価格、タスク分割、ノード割り当て、検証、監査ログ、支払い、報酬分配が別々の責務として動きます。

最初から巨大な P2P ネットワークを作るのではなく、まず中央の注文計画エンジンを確定させ、その後に API、Worker、検証、決済へ広げます。

## Phase 1: 注文計画エンジン

目的は、注文を受けて「どの処理を、どの単位に分け、どの種類のノードへ、どの検証条件で渡すか」を決めることです。

実装範囲:

- 注文モデル
- ノードモデル
- ワークロード別の処理プロファイル
- micro-yen 単位の価格見積もり
- チャンク分割
- compute / verification / canary タスク生成
- ノードスコアリング
- タスク割り当て
- 監査イベントのハッシュチェーン

このフェーズでは、実際のファイル転送、サンドボックス実行、決済、外部 API はまだ行いません。後続の実装が使う「計画結果」を安定した型で返すことに集中します。

完了条件:

- 1 件の注文から複数タスクを生成できる
- ノード一覧から割り当てを作れる
- 検証タスクが同一ノードに戻らない
- canary タスクを混ぜられる
- 監査ログの改ざん検出ができる
- CLI とテストで再現できる

## Phase 2: API と永続化

目的は、注文計画エンジンを外部から呼び出せる状態にすることです。

実装候補:

- `POST /orders`
- `GET /orders/:orderId`
- `GET /orders/:orderId/tasks`
- `POST /nodes/register`
- PostgreSQL などへの注文・タスク・割り当て保存
- S3 互換ストレージやローカルストレージへの入力ファイル保存
- 注文状態の状態遷移

状態例:

- `submitted`
- `priced`
- `planned`
- `queued`
- `running`
- `verifying`
- `completed`
- `failed`
- `disputed`

現在の実装:

- Node 標準 HTTP による API
- JSON ファイルストアによる注文計画・ノード永続化
- メールアドレス magic link によるサインイン
- セッション token と API key による Bearer 認証
- API key のハッシュ保存
- `.dispro/state.json` への保存
- `POST /orders` で注文計画を即時生成・保存
- `GET /orders/:orderId/tasks` で Worker 向けタスク情報を取得可能
- `GET /orders/:orderId/audit` で監査ログのハッシュチェーンを検証
- `POST /nodes/register` で `supportedWorkloads: ["*"]` の汎用ノード登録に対応

この段階では、状態遷移はまだ単純な `planned` 相当です。Phase 3 でタスクリース、実行中、結果提出、検証中、完了の状態を追加します。

## Phase 3: Worker プロトコル

目的は、実ノードがタスクを受け取り、処理結果を返せるようにすることです。

実装候補:

- ノード登録
- ハートビート
- タスクリース
- 結果アップロード
- 実行ログ送信
- ノード署名
- タスクのタイムアウトと再配布
- Worker SDK / CLI

最初は完全な P2P ではなく、中央 API からタスクを pull する形で始めるのが現実的です。

## Phase 4: 検証・不正対策

目的は、正しさを毎回全件再計算せずに担保することです。

実装候補:

- ランダム照合
- 多数決
- 罠タスク
- 既知答えタスク
- ノード信頼度更新
- 不一致時の追加検証
- 報酬没収
- 不正履歴の蓄積

検証ポリシーは、注文の金額、優先度、検証保証レベル、ノード信頼度によって変えます。

## Phase 5: 決済・報酬台帳

目的は、超小口注文と協力者報酬を内部台帳で扱えるようにすることです。

実装候補:

- 内部残高
- micro-yen 単位の台帳
- 仮報酬と確定報酬
- 手数料 5%
- 決済プロバイダ連携
- 出金申請
- 不正時の報酬取消

外部決済は円単位でも、内部台帳は 1 円未満を扱えるようにします。

## Phase 6: 本番ネットワーク化

目的は、個人・開発者・企業が使えるサービスへ拡張することです。

実装候補:

- Web UI
- API キー
- SDK
- GitHub Actions 連携
- 企業向け高信頼ノードプール
- GPU タスク
- リージョン・データ保持ポリシー
- メトリクス・監視
- セキュアサンドボックス
